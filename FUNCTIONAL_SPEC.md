# 나라장터 시장 인텔리전스 대시보드 기능명세서

> AI 에이전트와 개발자가 프로젝트의 공통 목표, 탭별 데이터 소스, 분석 기준, 구현 현황을 한 곳에서 확인하기 위한 기준 문서다.

- 문서 기준일: 2026-07-13
- 현재 UI: React 19 + Vite, 6개 탭
- API 프록시: Cloudflare Worker
- 배치 수집기: `pipeline/` Playwright + SQLite + JSON snapshot
- 구현 상태 표기: `구현` / `부분 구현` / `미구현`

## 1. 문서의 위치와 사용 규칙

새 작업을 시작하는 AI 에이전트는 다음 순서로 확인한다.

1. 이 문서에서 제품 목표와 탭별 데이터 경계를 확인한다.
2. 실제 동작은 항상 현재 코드로 재확인한다.
3. 기능·API·계산식·상태가 바뀌면 코드와 함께 이 문서를 갱신한다.
4. `README.md`는 실행 안내, `pipeline/README.md`는 배치 수집기 상세 안내로 사용한다.
5. DOCX 명세서는 제품 의도의 배경 자료다. DOCX와 현재 코드가 다르면 이 문서의 `현재 구현` 설명과 코드를 우선한다.

## 2. 제품 목표

제일테크와 동광프리즘이 공공조달 시장에서 다음 질문에 답하도록 한다.

- 지금 참여할 수 있는 공고는 무엇인가?
- 공고 전 발주계획과 사전규격을 통해 어디에 선제 영업해야 하는가?
- 지방자치단체의 예산이 얼마나 남았고 집행이 얼마나 지연되었는가?
- 경쟁사는 어디서 얼마에 낙찰받고 어떤 투찰률 구간에 집중하는가?
- 발주처는 특정 업체를 반복 선택하는가, 신규 진입이 가능한가?
- 예산, 집행 지연, 경쟁도, 발주계획, 사전규격을 합쳤을 때 어느 기관을 우선 공략해야 하는가?

핵심 차별점은 `입찰공고` 단계만 보지 않고 `예산 → 발주계획 → 사전규격 → 입찰 → 낙찰·계약`의 전체 흐름을 영업 의사결정으로 연결하는 것이다.

## 3. 회사별 검색 범위

### 3.1 제일테크

- 일반 API 키워드: `흡연부스`, `이동식초소`, `흡연실`, `부스`
- 예산 API 키워드: `흡연부스`, `흡연실`, `흡연시설`, `초소`, `자전거거치대`, `자전거보관소`

### 3.2 동광프리즘

- 일반 API 키워드: `분리수거함`, `분리수거`, `쓰레기통`, `클린하우스`, `제설함`, `음식물쓰레기통`, `쓰레기수거함`
- 예산 API 키워드: `클린하우스`, `클린 하우스`, `분리수거`, `재활용정거장`, `분리배출시설`, `음식물`, `제설함`, `쓰레기`

사용자가 일반 키워드를 추가·삭제할 수 있으며 `localStorage`에 즉시 저장한다. Worker에 `DASHBOARD_CACHE` KV가 연결된 경우 `/api/pipeline-keywords`에도 회사별 목록을 동기화하며, pipeline은 월간 실행 시 이 공용 목록을 사용한다. 키워드가 0개면 해당 회사의 일반 API를 호출하지 않는다.

## 4. 시스템과 데이터 흐름

```text
React/Vite frontend (:5174)
  ├─ GET Worker?endpoint=...&keyword=...
  │    └─ data.go.kr 조달청 API / 지방재정365
  └─ GET /delivery.json (탭 4 진입 시 지연 로드)
       └─ pipeline 이 생성한 정적 snapshot

pipeline
  조달데이터허브 WebSquare/MicroStrategy 보고서
    → Playwright 그리드 추출
    → SQLite upsert
    → frontend/public/delivery.json
```

`delivery.json`은 탭 4에 처음 진입할 때 한 번 로드한다. snapshot의 `keywords`와 납품요구명·품목명을 현재 회사의 일반+예산 키워드와 매칭해 회사별 표시 범위를 나눈다.

Worker는 API 키를 숨기고 endpoint별 파라미터 변환과 응답 필드 정규화를 담당한다. 프론트엔드는 탭 진입 시 필요한 endpoint만 지연 로드한다.

## 5. 탭별 기능명세

### 탭 1. 공고 모니터링 + 투찰 분석

**목적**

선택 회사의 품목과 관련된 입찰공고 중 아직 개찰일이 지나지 않은 공고를 빠른 순서로 보여준다.

**API와 조회 기준**

- endpoint: `bids`
- 조달청 입찰공고정보서비스 `BidPublicInfoService`
- operation: `getBidPblancListInfoThngPPSSrch`
- 검색 필드: `bidNtceNm={키워드}`
- 기간: 최근 30일
- 페이지: 키워드당 100건 × 최대 2페이지

**가공과 표시**

- 공고번호별 최신 `bidNtceOrd`만 남긴다.
- 최신 차수의 `ntceKindNm`에 `취소`가 있으면 해당 공고 전체를 제거한다.
- `opengDt`가 오늘 이후인 항목만 남기고 개찰 D-day 오름차순으로 정렬한다.
- 공고명, 기관, 추정가격/배정예산, 개찰일, 마감일, 참가가능지역, 상세 링크와 `투찰 분석` 버튼을 표시한다.

**공고별 투찰 분석 패널**

- 공고 행에서 버튼을 누르면 페이지 이동 없이 우측 패널을 연다.
- 선택 공고의 기관, 품목, 기초금액, 개찰일을 자동 전달하고 `awards`를 지연 로드한다.
- `awards` 조회 기본 기간은 최근 3년이며 키워드당 100건 × 최대 5페이지다.
- 기관명 유사 매칭 + 공고/품목명의 의미 토큰 매칭 + 최근 3년 조건으로 기관 이력을 만든다.
- 낙찰률은 `낙찰금액 / 예정가격 × 100`으로 계산하고 0.1%p 히스토그램, 평균·중앙값·최저·최고를 표시한다.
- 기관 표본 10건 이상은 기관 분포만, 5~9건은 기관·품목 전체를 병행, 1~4건은 품목 전체로 대체, 0건은 `이 기관 첫 진입`으로 표시한다.
- 사용자가 입력한 하한율을 기준으로 공격 `max(하한+0.005, 최저)`, 표준 `max(하한+0.02, 중앙값)`, 안전 `max(하한+0.05, 평균)`을 계산하고 기초금액 기준 원 단위 예상금액을 표시한다.
- 경쟁사 평균이 권장 범위 안이면 해당 평균 바로 아래이되 하한보다 높은 구간을 자동 코멘트로 제안한다.
- 자사 이력은 패널 하단에서 입력하며 `g2b-my-bids`에 회사·기관·공고별로 저장한다.
- 낙찰정보서비스는 낙찰자 결과 중심이므로 경쟁사 영역은 전체 참여업체가 아닌 해당 기관·유사 품목에서 확인된 낙찰 경쟁사다.

**현재 상태: 구현**

제목은 `오늘 입찰 가능한 공고`지만 실제 필터는 입찰마감일이 아니라 개찰일 기준이다. 패널의 3년 분석도 API 페이지 상한 안에서 수집된 표본을 사용한다.

### 탭 2. 발주계획·사전규격

**목적**

입찰공고보다 앞선 발주계획과 사전규격 단계에서 영업 기회를 포착하고, 카탈로그 발송·사전규격 의견 제출 타이밍을 제공한다.

**API 1: 발주계획**

- endpoint: `plan`
- `OrderPlanSttusService/getOrderPlanSttusListThngPPSSrch`
- 검색: `bizNm={키워드}`
- 조회 등록 범위: 최근 90일
- 발주 예정월: 현재 월부터 3개월 뒤까지
- 표시: 사업/품목명, 수요기관, 발주예정금액, 발주예정일, 공고기관, 선제 영업 메모

**API 2: 사전규격**

- endpoint: `spec`
- `HrcspSsstndrdInfoService/getPublicPrcureThngInfoThngPPSSrch`
- 검색: `prdctClsfcNoNm={키워드}`
- 기간: 최근 14일
- 표시: 품목명, 기관, 공개일, 규격 요약, 의견등록 마감 D-day, 문서/상세 링크

**공통 페이지**

- 키워드당 100건 × 최대 2페이지

**현재 상태: 구현**

사전규격의 마감 경과 항목을 별도로 제거하지는 않는다. 월 단위로만 제공되는 발주예정일은 정확한 D-day로 해석하지 않는다.

### 탭 3. 지역 예산·집행률

**목적**

지방자치단체의 현재 예산현액과 지출액을 기준으로, 예산이 크게 남았고 집행이 느린 사업을 영업 우선순위로 제시한다.

**API와 조회 기준**

- endpoint: `lofin`
- 지방재정365 세부사업별 세출현황
- 실제 API 경로: Worker 환경변수 `LOFIN_API_PATH`
- 인증키: `LOFIN_KEY`
- 검색: `dbiz_nm={회사별 예산 키워드}`
- 회계연도: 현재 KST 연도
- 집행일자: 어제부터 최대 7일 전까지 조회해 데이터가 처음 있는 snapshot 선택
- 페이지: 키워드당 100건 × 최대 3페이지
- 보조 endpoint: `plan` (해당 기관의 발주계획 유무 대조)

**핵심 계산**

```text
집행잔액 = 예산현액 - 지출액
집행률 = 지출액 / 예산현액 × 100
연중 경과율 = (오늘 - 1월 1일) / (12월 31일 - 1월 1일) × 100
지연지수 = 연중 경과율 - 집행률
```

**영업 등급**

- `즉시 영업`: 지연지수 40 이상 AND 잔액 3,000만 원 이상
- `적극 접촉`: 지연지수 20 이상 AND 잔액 1,000만 원 이상
- `보류`: 지연지수 0 미만 OR 잔액 500만 원 미만
- `관찰`: 나머지

**필터와 표시**

- 1차 영업: 예산현액 내림차순
- 2차 영업: 잔액 1,000만 원 이상, 지연지수 내림차순
- 연말 소진: 집행잔액 내림차순
- 전체: 지연지수 내림차순
- 지역, 미접촉, 물품 사업만 필터
- 사업명의 물품/용역 신호를 휴리스틱으로 분류
- 기관·사업별 영업 상태와 메모는 `localStorage`에 저장

**현재 상태: 구현**

`LOFIN_KEY`와 `LOFIN_API_PATH`가 없으면 탭에 데이터가 나오지 않는다.

### 탭 4. 경쟁사 분석

**목적**

선택 품목의 최근 낙찰 결과를 업체별로 집계해 주요 경쟁사와 수주 규모, 주요 기관을 보여준다.

**API와 조회 기준**

- endpoint: `awards`
- 조달청 낙찰정보서비스 `ScsbidInfoService`
- operation: `getScsbidListSttusThngPPSSrch`
- `inqryDiv=2`
- `bidNtceNm={키워드}`
- `inqryBgnDt={3년 전 0000}`
- `inqryEndDt={오늘 2359}`
- `numOfRows=100`, 최대 5페이지

**집계**

- 업체명: `sucsfbidCorpNm` 우선
- 수주건수: 업체별 낙찰 행 수
- 수주금액: `sucsfbidAmt` 합계
- 낙찰률: `sucsfbidAmt / presmptPrce × 100`; 분모가 없으면 API 낙찰률 필드 사용
- 주요 기관: 해당 업체의 낙찰 건수가 가장 많은 기관
- 정렬: 총 수주금액 내림차순, Top 10

**종합쇼핑몰 납품요구 데이터**

- `pipeline/`이 `frontend/public/delivery.json`을 생성하고 탭 4가 지연 로드한다.
- 납품업체별 MAS 판매(납품요구) 건수, 품목 수, 납품금액, 거래기관 수, 주요 수요기관을 집계한다.
- 낙찰정보와 납품요구를 업체명 정규화 후 통합 비교표에 보이되, 낙찰금액과 납품금액은 별도 컬럼으로 유지한다.
- `나라장터 낙찰 경쟁사 Top10`과 `MAS 판매 Top10`을 출처별 카드·차트로 표시한다.
- 서로 다른 실적이므로 낙찰금액과 납품금액을 합산한 순위는 만들지 않는다.

**현재 상태: 구현**

로컬 snapshot 기준 낙찰·납품 통합 비교는 구현됨. snapshot 자동 배포는 별도 운영 과제다.

### 탭 5. 발주처 패턴

**목적**

수요기관별 계약 업체 반복 선택 정도를 계산해 특정 업체의 텀세가 강한지, 신규 진입이 가능한지 판단한다.

**API와 조회 기준**

- endpoint: `contract`
- 조달청 계약정보서비스 `CntrctInfoService`
- operation: `getCntrctInfoListThngPPSSrch`
- 검색: `prdctClsfcNoNm={키워드}`
- 기간: 최근 30일
- 페이지: 키워드당 100건 × 최대 2페이지

**집계와 판정**

```text
기관별 계약금액 = 계약금액 합계
기관별 계약건수 = 계약 행 수
HHI = Σ(업체별 계약건수 / 해당 기관 전체 계약건수)²
```

- `텀세 주의`: HHI 0.72 이상 AND 계약 3건 이상
- `진입 추천`: HHI 0.45 이하 AND 계약금액 > 0
- `관찰 필요`: 나머지
- 계약금액 내림차순 Top 15

**현재 상태: 구현**

현재 분석 기간이 30일이므로 장기 반복 발주 패턴으로 일반화할 때는 주의한다.

### 탭 6. 공략 기회

**목적**

계약 규모, 예산 집행 지연, 경쟁 집중도, 발주계획, 사전규격을 합산해 우선 공략할 기관을 자동 추천한다.

**사용 endpoint**

- `contract`: 후보 기관, 기관별 계약금액, HHI, 계약 품목
- `lofin`: 기관별 최대 지연지수
- `plan`: 발주계획 유무
- `spec`: 사전규격 유무
- `price`: 계약 품목과 유사한 가격정보 건수

**공략 점수**

```text
계약금액 상대 규모              최대 25점
지연지수(0~50 → 0~25)           최대 25점
독점도가 낮은 정도 (1 - HHI)       최대 20점
발주계획 존재                         20점
사전규격 존재                         10점
```

- 80점 이상: `적극 공략`
- 60~79점: `검토 필요`
- 60점 미만: `진입 어려움`
- 점수 내림차순 Top 12

**중요한 해석 주의**

- 화면의 `예산`은 지방재정365 예산현액이 아니라 **최근 30일 기관별 계약금액 합계**다.
- 후보 모집단은 탭 5의 Top 15 기관이다.
- 지방재정 기관 대조는 유사 기관명 매칭을 사용한다.
- 발주계획·사전규격 가점은 현재 기관명 정확 일치를 사용하므로 표기 차이로 가점이 누락될 수 있다.
- `price`는 점수에 직접 가점하지 않고 화면에 유사 가격정보 건수만 표시한다.

## 7. 캐시와 로딩 규칙

- 초기 로드는 탭 1의 `bids`만 요청한다.
- 나머지 endpoint는 탭 진입 시 필요한 것만 요청한다.
- 프론트엔드 `localStorage` 캐시 TTL은 24시간이다.
- 캐시 키는 회사와 키워드 목록을 포함한다.
- 현재 요청 중인 endpoint는 `fetchingEndpoints`로 별도 추적하여, 이전 데이터가 화면에 남아 있어도 새로고침 상태를 표시한다.
- 기본 자동 로드에서 이미 실패로 기록된 endpoint는 무한 재시도하지 않는다. 사용자 새로고침으로 재시도한다.

## 10. 지역 분류 원칙

- 전국 단위 공공기관을 시도 이름보다 먼저 검사한다. 예: `인천국제공항공사`를 인천으로 오분류하지 않는다.
- 시도 정식명칭·약칭을 매칭한다.
- 시군구만 있는 기관명은 역매핑 테이블로 시도를 찾는다.
- 지방재정365가 `lofinRegionNm` 필드를 제공하면 기관명 파싱보다 우선한다.
- 판단할 수 없으면 `미분류`로 남긴다.

## 11. 수정 시 보존해야 할 불변 규칙

1. 회사별 키워드, 캐시, 자사 투찰 이력, 영업 메모를 혼합하지 않는다.
2. 날짜·금액·기관명의 API별 차이는 Worker/변환 계층에서 흡수한다.
3. 탭은 필요한 endpoint만 지연 로드하고, 한 endpoint 실패가 기존 성공 데이터를 지우지 않게 한다.
4. 캐시·실시간·만료 데이터를 UI에서 구분한다.
5. 납품요구 실적과 낙찰 실적은 출처와 의미가 다르므로 원본 구분을 보존한다.
6. 납품요구 데이터로 투찰률을 임의 계산하지 않는다.
7. 표시 기준·가중치·기간·API 파라미터를 바꾸면 이 문서를 같이 수정한다.
